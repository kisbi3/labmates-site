00 · The workbench

내 연구 전체를 한 화면에서 읽습니다.

노트, 코드, 수식, Git, 작업 보드와 연구 에이전트가 하나의 프로젝트 경계 안에서 이어집니다.

01 · How it works

연구 폴더를 여는 순간부터, 승인된 기록이 남기까지.

기존 파일을 가져오거나 전용 형식으로 바꾸지 않습니다. Asterelle은 현재 워크스페이스를 읽고, 다음 판단까지 이어지는 흐름을 만듭니다.

어느 연구자의 아침

지난주에 멈춘 분석 프로젝트를 엽니다. Observatory가 최근 변화와 야간 검증 결과를 모아 보여줍니다. 연구자는 Git 근거를 확인하고, 필요한 후속 작업을 Mates에 맡긴 뒤, 신뢰한 결과만 프로젝트 기록으로 승격합니다.

01 / Open

기존 연구 폴더를 엽니다.

Markdown, 코드, 데이터와 Git 저장소는 원래 위치에 남습니다. Asterelle은 프로젝트 경계와 연결을 읽습니다.

02 / Observe

다음 판단을 확인합니다.

Observatory에서 프로젝트 변화, 막힌 작업, 야간 결과와 결정 대기 항목을 근거와 함께 봅니다.

03 / Work

직접 작업하거나 맡깁니다.

파일, 노트북과 터미널에서 이어서 작업하거나 Mates에 프로젝트 범위가 정해진 일을 위임합니다.

04 / Review

결과와 근거를 함께 봅니다.

제안된 변경, 실행 기록, 파일과 Git 근거, 작성자와 검증 상태를 하나의 검토 항목에서 확인합니다.

05 / Ratify

신뢰한 것만 승격합니다.

승인한 후보만 정본 연구 상태로 들어갑니다. 기각과 보류도 판단 이력으로 남습니다.

02 · One research system

도구를 모은 것이 아니라, 연구 흐름을 연결했습니다.

  1. Constellation무엇이 무엇과 닿아 있는지 봅니다.

    Markdown에 이미 존재하는 링크에서 논문, 개념, 방법, 주장과 프로젝트의 관계를 그립니다. 인덱스는 다시 만들 수 있고, 원본 노트는 평범한 파일로 남습니다.

  2. Files & Workbench읽는 곳과 실행하는 곳을 가깝게 둡니다.

    Scientific Markdown, 코드, 노트북, 터미널, Git과 작업 보드가 같은 프로젝트 경계를 따릅니다. 로컬과 SSH 워크스페이스에서도 범위가 화면에 드러납니다.

  3. Mates역할과 관계가 보이는 연구 동료들.

    프로젝트 에이전트는 명시된 위임 경로를 따라 준비하고 검토하고 검증합니다. 작업 진행과 실행 기록은 대화가 끝난 뒤에도 남습니다.

  4. Observatory오늘 판단할 일이 한곳에 도착합니다.

    모든 프로젝트의 신호, 야간 결과와 연결 후보를 근거와 함께 모아 보여줍니다. 어떤 결과도 조용한 자동 편집으로 들어오지 않습니다.

03 · One environment, many models

연구의 각 단계에 알맞은 모델을 사용합니다.

이미 사용 중인 모델 접근 권한을 가져오세요. Asterelle은 역할별 모델을 하나의 프로젝트 맥락에 연결하고, 모든 결과에 같은 근거와 검토 규칙을 적용합니다.

01 / BRING

이미 가진 모델을 사용합니다.

기존 Claude Code·Codex 접근 권한과 설정된 로컬·원격 provider를 연결해 하나의 모델 회사에 연구 흐름 전체를 맡기지 않습니다.

02 / ASSIGN

역할별로 모델을 배치합니다.

조율, 코딩, 독해, 요약, 야간 준비와 독립 검증에 서로 다른 모델을 사용할 수 있습니다.

03 / CONTINUE

프로젝트 기억은 이어집니다.

모델을 바꿔도 작업, 실행 기록, 근거 참조와 프로젝트 경계는 사라지지 않습니다.

04 / CONTROL

비용과 locality를 통제합니다.

반복적이거나 민감한 준비 작업은 설정에 따라 로컬에 두고, 필요한 작업에만 강한 원격 모델을 사용합니다.

모델 가용성과 데이터 처리는 연구자가 설정한 provider와 locality 정책을 따릅니다. fallback 결과가 정본 연구 기록으로 자동 승격되지는 않습니다.

04 · How it earns trust

마지막 판단은 언제나 연구자의 몫입니다.

  1. 01 에이전트는 제안하고, 연구자는 승격합니다.연구 상태를 바꾸는 작업은 명시적인 게이트를 통과하며, 에이전트가 최종 권한을 갖지 않습니다.
  2. 02 작성자와 검증자를 분리합니다.독립된 기준과 역할이 산출물 작성과 최종 판단을 구분합니다.
  3. 03 연구의 정본은 프로젝트 안에 남습니다.Markdown이 정본이고, 그래프와 검색 인덱스는 다시 만들 수 있는 파생 상태입니다.
  4. 04 사람이 없는 작업은 선택 사항입니다.야간 자동화는 설정된 한도 안에서 대기열의 작업만 실행하고, 검토할 근거를 돌려줍니다.

05 · Private beta

당신의 연구 흐름에서 시험해 보세요.

Asterelle은 소수의 개인 연구자와 실제 연구 워크스페이스에서 테스트하고 있습니다. 신청하면 사용 환경을 확인한 뒤 적합한 데스크톱 빌드와 시작 방법을 안내합니다.

  • macOS · Windows · Linux
  • 기존 Claude Code 또는 Codex 접근 권한 사용
  • 시스템 Python을 변경하지 않음

06 · FAQ

연구 맥락을 맡기기 전에 확인할 것들.

Obsidian에 AI 플러그인을 붙이는 것과 무엇이 다른가요?
Obsidian은 노트를 연결합니다. Asterelle은 그 관계를 프로젝트별 코드, 작업 보드, 지속되는 에이전트 실행과 명시적인 연구 게이트까지 이어줍니다.
에이전트가 연구 기록을 마음대로 바꿀 수 있나요?
아니요. 에이전트 산출물은 후보로 남으며, 에이전트 스스로 발급할 수 없는 명시적 확인을 거쳐야 정본으로 승격됩니다.
어떤 정보가 컴퓨터 밖으로 나가나요?
모델 호출은 사용자가 설정한 provider와 워크플로의 문맥 경계를 따릅니다. 연구 상태는 워크스페이스에 남고, 야간 작업의 모델과 locality 설정도 배포 설정에서 선택합니다.
실행하려면 무엇이 필요한가요?
연구 워크스페이스와 Claude Code 또는 Codex 접근 권한이 필요합니다. Asterelle은 시스템 Python을 수정하지 않고 제품 전용 디렉터리에 런타임을 구성합니다.
기존 Obsidian vault를 그대로 사용할 수 있나요?
네. Markdown을 원래 위치에서 인덱싱하고 wiki-link를 그래프 관계로 해석합니다. 파일은 다른 편집기에서도 계속 읽을 수 있고, 파생 인덱스는 다시 만들 수 있습니다.
언제 사용할 수 있나요?
현재 직접 배포 방식의 private beta를 진행하고 있습니다. 사용 환경을 알려주시면 적합한 빌드가 준비됐을 때 연락드리겠습니다.

더 확인하고 싶은 점이 있나요? 직접 문의해 주세요.
